# 设计模式标注规范

## 触发条件
写任何设计文档、架构方案、重构 plan 时必须执行。

---

## 规则

### 1. 内联标注（代码块内）

每处用到设计模式的地方，在代码注释里标明：

```cpp
// [Strategy] 抽象策略：规定接口契约，不含任何算法实现
class ITrajectoryPlanner { ... };

// [Facade] 对外一个入口，内部封装 5 步流程
Result<Trajectory> Plan(...);

// [Pimpl 精神] 依赖三方类型的 helper 放匿名 namespace，头文件零三方类型
namespace { ... }
```

### 2. 独立总结节（文档末尾）

每份设计文档末尾必须有 `## 设计模式说明` 节，每个模式写清楚：

| 字段 | 说明 |
|---|---|
| **什么时候用** | 触发这个模式的场景特征 |
| **本次怎么用** | 结合当前代码的具体结构图或示例 |
| **好处** | 这个选择带来了什么 |
| **缺点 / 注意事项** | 这个选择的代价或潜在问题 |
| **识别信号** | 以后看到什么代码特征就能认出这个模式 |

### 3. 设计决策表里也要体现

在 `## 设计决策` 表的"原因"列里，如果用到了某个模式，注明模式名：

| 决策点 | 选择 | 原因 |
|---|---|---|
| 返回类型 | `Result<T>` | **[Result 惯用法]** 调用方被迫处理失败，错误显式传递 |
| 接口抽象 | `ITrajectoryPlanner` | **[Strategy]** 算法可替换，调用方不依赖具体实现 |

---

## 常用模式速查

| 模式 | 识别信号 | 好处 | 缺点 |
|---|---|---|---|
| **Strategy** | `class IXxx { virtual Xxx() = 0; }` + 多实现类 | 算法可替换，符合开闭原则 | 多一层间接调用；小项目过度设计 |
| **Facade** | 一个 public 方法内顺序调多个 private 步骤 | 调用方只关心结果，不关心步骤 | 隐藏细节，调试时需深入看内部 |
| **Pimpl** | 头文件 `struct Rep;` + cpp 里才定义 `struct Rep` | 头文件隔离三方依赖，加速编译 | 多一次堆分配；不能内联成员函数 |
| **Pimpl 精神（匿名 ns）** | 私有 helper 下沉到 `.cpp` 匿名 namespace | 零额外开销，头文件干净 | 只适合自由函数，不适合需要状态的场景 |
| **Result 惯用法** | 返回 `Result<T>`，调用处 `if (!result)` | 错误显式、可组合、无异常开销 | 调用链长时传递繁琐（可用 `?` 运算符缓解） |
| **Observer / 信号槽** | Qt `signals` / `connect` | 解耦发送方和接收方 | 运行时连接，调试难追踪，注意防循环 |
| **Repository** | `IXxxRepository` + 内存/文件/DB 实现 | 持久化可替换，业务逻辑不依赖存储细节 | 增加抽象层，简单场景过度设计 |
